12-Month Plan Holmes — 8-Week Sprint
North star
Tailored to founding/early AI-engineer roles building agentic QA — systems that reason about product intent, drive a real browser, and get measurably sharper every release (the Holmes job post is the reference brief). The job is multi-provider by design: every phase below treats Claude, GPT, Gemini, Qwen, GLM, Kimi, and Gemma as interchangeable parts you route between, not a single stack you commit to.
The stack you're mastering
Multi-Agent Architecture
Reasoning ↔ Planning layer
Planner, executor, critic, reporter. The core skill: decomposing "does this product work as intended" into agents that reason about intent instead of running a fixed script.
Browser Automation
Agent ↔ Browser layer
Stagehand, BrowserUse, Playwright MCP. Natural-language action over a real UI, resilient to the UI changing underneath it — no selectors to maintain by hand.
Multi-LLM Routing
Model ↔ Task layer
Claude, GPT, Gemini, Qwen, GLM, Kimi, Gemma. Pick per agent role on cost, latency, and quality — frontier models for ambiguous reasoning, cheap open-weight models for repetitive execution.
Evals & Benchmarks
Quality ↔ Feedback layer
Regression suites that prove the system got sharper, not just different, after every prompt or model change. Treated as a build artifact, not an afterthought.

Build mode (desk)
Project 1A — Provider bake-off
Pick one structured QA-relevant task (e.g. extract a structured bug report from a transcript, or classify pass/fail from a log) and run it through Claude, GPT, Gemini, and at least one open-weight model (Qwen, GLM, or Kimi) via PydanticAI's model-agnostic interface. Log cost per call, latency, and a quality score for each. This table is the actual deliverable — it's the skill the job is hiring for.
Project 1B — Tool-use & context-window stress test
One agent, 3–4 tools, a deliberately long multi-step tool chain and a large tool-result payload. Compare how each provider handles error retries, parallel tool calls, and context truncation as you approach its context-window limit.
⬡ Architecture insight
There is no single "best" model in 2026 — there's a best model per role. Frontier closed models (Claude, GPT, Gemini) tend to win on ambiguous product-intent reasoning; open-weight models (Qwen, GLM, Kimi, Gemma) tend to win on cost and latency for repetitive, well-specified steps. The job explicitly asks you to balance this per agent — Project 1A's bake-off table is your evidence that you can.
Phase 1 exits when you can...

◻Name a concrete cost/latency/quality tradeoff between at least 4 providers for a specific agent role
◻Swap the underlying model for a running agent without rewriting its tool or output-schema code

Build mode (desk)
Project 2A — Same flow, three tools
Implement one multi-step UI flow (sign-up + a 3-step task on a public demo site) three times: once in Stagehand, once in BrowserUse, once via an MCP agent wired to Playwright MCP. Document where each one breaks and where it falls back to raw Playwright.
Project 2B — Resilience test
Change the UI underneath each implementation (rename a button, reorder a step, add a modal) and measure which one self-heals via natural-language re-grounding versus which one breaks like a brittle Selenium script. This is literally Holmes' pitch — "no scripts, no manual maintenance" — prove it to yourself first.
⬡ Architecture insight
The real design decision isn't "which tool" — it's where you draw the line between letting the model decide each step (slow, expensive, most resilient) and giving it a typed action to execute (fast, cheap, brittle if the UI changes underneath it). Stagehand, BrowserUse, and Playwright MCP each draw that line in a different place; you'll need to draw it deliberately per agent, not by default.
Phase 2 exits when you can...
◻Drive a real multi-step browser flow with each of Stagehand, BrowserUse, and Playwright MCP
◻Explain how each framework grounds a natural-language instruction into an actual selector or accessibility node
◻Demonstrate at least one case where an agent recovered from a UI change without code changes

Build mode (desk)
Project 3A — Multi-agent test generator
Given a plain-English feature description (no scripts), a planner agent proposes a test plan, your Phase 2 browser-agent executes it, and a critic agent judges pass/fail against the original intent — not against a hardcoded assertion. This is the core Holmes loop.
Project 3B — Eval harness
Build 15–20 (feature description, app, known-correct outcome) triples. Score Project 3A's accuracy against them, run the whole harness across at least 2 LLM providers from Phase 1, and keep it as a regression suite you re-run on every prompt or model change.
⬡ Architecture insight
"Reasons about product intent" means the critic agent has to judge against a spec, not a literal string match — which makes your eval set itself an agentic-design problem, not a testing afterthought. The benchmarks above (WebArena, tau-bench) solve this the same way you will: simulate the user/spec side and score outcomes, not steps.
Phase 3 exits when you can...
◻Run a planner → executor → critic loop where the critic judges against natural-language intent, not a fixed assertion
◻Build a reusable eval set and explain its precision/recall tradeoffs for "did this feature work"
◻Show a regression your eval suite caught when you swapped a model or changed a prompt
