Step 2 — Define the tools with @tool

Same logic as 1A's tools, but the docstring is now the entire tool-selection contract Strands hands the model — there's no separate Pydantic Field(description=...) layer:

# strands_version/research_agent.py
import httpx
import trafilatura
from duckduckgo_search import DDGS
from strands import Agent, tool
from strands.models import BedrockModel


@tool
def web_search(query: str, max_results: int = 5) -> list[dict]:
    """Search the web for a query. Returns a list of dicts with
    'title', 'href', and 'body' keys, ranked by relevance."""
    with DDGS() as ddgs:
        return list(ddgs.text(query, max_results=max_results))


@tool
def fetch_page(url: str) -> str:
    """Fetch a URL and extract its main readable text content,
    discarding navigation, ads, and boilerplate. Returns plain text,
    or an empty string if extraction fails."""
    resp = httpx.get(
        url, timeout=10.0, follow_redirects=True,
        headers={"User-Agent": "research-agent/0.1 (+https://staycurious.me)"},
    )
    resp.raise_for_status()
    text = trafilatura.extract(resp.text, include_comments=False, include_tables=False)
    return text or ""


_summarizer = Agent(
    model=BedrockModel(model_id="us.anthropic.claude-haiku-4-5-20251001-v1:0"),
    system_prompt=(
        "Condense the given document into a factual 3-5 sentence "
        "summary. No opinions, no filler."
    ),
)


@tool
def summarize_source(text: str) -> str:
    """Summarize a single fetched document's raw text into a short
    factual digest suitable for downstream synthesis."""
    result = _summarizer(text[:12000])
    return str(result)
Ergonomics note: Strands' @tool needs nothing beyond a type-hinted function and a docstring — no separate schema class, no provider wiring inside the tool itself. That's less code than PydanticAI's tool registration, but you also get no structural validation of the tool's own inputs beyond what Python type hints imply (Strands does generate a JSON schema from the signature for the model, but it isn't Pydantic-validated at call time).

Step 3 — Build the orchestrator agent

model = BedrockModel(model_id="us.anthropic.claude-sonnet-4-5-20251001-v1:0")

agent = Agent(
    model=model,
    system_prompt=(
        "You are a research assistant. Given a topic:\n"
        "1. Call web_search to find candidate sources.\n"
        "2. Call fetch_page on the top 3 most relevant results.\n"
        "3. Call summarize_source on each fetched page's text.\n"
        "4. Synthesize the summaries into a final answer.\n\n"
        "Respond with ONLY a JSON object matching this exact shape, "
        "no markdown fences, no extra prose:\n"
        '{\n'
        '  "summary": "150-250 word synthesis",\n'
        '  "key_findings": ["finding 1", "finding 2", "finding 3"],\n'
        '  "sources": ["https://...", "https://..."],\n'
        '  "confidence": "low" | "medium" | "high"\n'
        "}"
    ),
    tools=[web_search, fetch_page, summarize_source],
)
This is the crux of the exercise: Strands has no output_type parameter. The JSON contract lives entirely in prose, inside the system prompt. The model can and will drift from it — which is exactly the gap you validate against manually in the next step.

Step 4 — Validate the output manually (Strands has no native retry loop)

Reuse the ResearchBrief Pydantic model from 1A purely as a validator — you're not using PydanticAI's agent machinery, just its schema class:

import json
import re
import sys

sys.path.insert(0, "../pydantic_version")
from research_agent import ResearchBrief  # reuse the schema, not the agent

from pydantic import ValidationError


def _extract_json(raw_text: str) -> dict:
    """Strands returns free-form text; strip markdown fences if present
    and locate the first {...} block."""
    cleaned = re.sub(r"^```(?:json)?|```$", "", raw_text.strip(), flags=re.MULTILINE)
    match = re.search(r"\{.*\}", cleaned, re.DOTALL)
    if not match:
        raise ValueError(f"No JSON object found in model output:\n{raw_text}")
    return json.loads(match.group(0))


def run_with_validation(topic: str, max_attempts: int = 3) -> ResearchBrief:
    """Manual retry loop replicating PydanticAI's ModelRetry behavior:
    on validation failure, feed the error back to the model and ask
    it to correct its own output."""
    feedback = ""
    for attempt in range(1, max_attempts + 1):
        prompt = topic if attempt == 1 else (
            f"{topic}\n\nYour previous answer was invalid: {feedback}\n"
            f"Fix it and return ONLY the corrected JSON object."
        )
        raw = str(agent(prompt))
        try:
            data = _extract_json(raw)
            return ResearchBrief.model_validate(data)
        except (ValueError, ValidationError, json.JSONDecodeError) as exc:
            feedback = str(exc)
            if attempt == max_attempts:
                raise RuntimeError(
                    f"Failed to get valid ResearchBrief after {max_attempts} attempts: {feedback}"
                ) from exc
    raise AssertionError("unreachable")
This is the concrete answer to "how does Strands handle structured output differently": PydanticAI wires validation + retry into agent.run() itself. Strands treats the model output as plain text/tool-result data — you own parsing, validation, and the retry loop. More code, but full visibility into every retry decision.

Step 5 — CLI entry point

import argparse


def main() -> None:
    parser = argparse.ArgumentParser(description="Research agent (Strands rebuild)")
    parser.add_argument("topic", help='e.g. "quantum error correction 2026"')
    parser.add_argument("-o", "--output", help="Write JSON result to this file")
    args = parser.parse_args()

    brief = run_with_validation(args.topic)
    brief_json = brief.model_dump_json(indent=2)
    print(brief_json)

    if args.output:
        with open(args.output, "w") as f:
            f.write(brief_json)


if __name__ == "__main__":
    main()

Troubleshooting

AccessDeniedException from Bedrock: you haven't been granted access to the specific model ID in your region. Fix in Bedrock console → Model access, and double check the model ID string matches exactly (region prefix like us. matters for cross-region inference profiles).
Model output isn't valid JSON even after retries: Sonnet sometimes wraps JSON in markdown fences or adds a leading sentence despite instructions. Strengthen the regex in _extract_json, or add "Do not include any text before or after the JSON object" more forcefully in the system prompt, or lower temperature via BedrockModel(model_id=..., temperature=0.2).
ExpiredTokenException / credentials error: if using temporary AWS SSO credentials, they expire — rerun aws sso login and re-export session env vars before retrying.
Nested summarizer agent is slow: each summarize_source call is itself a full Bedrock round trip. For 3 sources this can add 10-20s. Consider asyncio.gather if you refactor to Strands' async invocation, or just budget for it in a synchronous CLI tool.
