Build Guide: Wire to MCP Servers

Project 1C companion guide · Builds on Project 1B (Strands) · MCP v1.0 (Nov 2025)

What You'll Build

  • The Strands research agent from Project 1B, with fetch_page and web_search replaced by real MCP servers instead of your own tool code
  • A Strands MCPClient wired to two MCP servers over STDIO transport: the official Fetch MCP server (page retrieval) and the Filesystem MCP server (local read/write, standing in for the "AWS Docs" style reference server)
  • A captured JSON-RPC message trace from MCP Inspector, saved as mcp_inspector_trace.json
  • A short section on the Nov 2025 MCP v1.0 browser-based OAuth flow for when a remote/hosted MCP server requires authentication
  • Exact deliverables:
    • mcp_version/research_agent.py — Strands agent consuming MCP tools
    • mcp_version/mcp_inspector_trace.json — captured JSON-RPC exchange
    • mcp_version/sample_output.json
    • README note on OAuth for remote servers

Architecture

research_agent.py (Strands, unchanged orchestrator/system prompt)
        │
        │ tools=[ ...MCP-discovered tools..., summarize_source ]
        │
        ▼
 ┌──────────────────────────────────────────────┐
 │ Strands MCPClient (one per server)            │
 │ transport: STDIO (subprocess pipes)           │
 └───────────────┬────────────────┬──────────────┘
                 │                │
                 ▼                ▼
     npx @modelcontextprotocol/  npx @modelcontextprotocol/
     server-fetch                server-filesystem /path
     (spawned subprocess)        (spawned subprocess)
                 │                │
        JSON-RPC 2.0 over stdin/stdout
                 │                │
     tools/list, tools/call       tools/list, tools/call
     ("fetch")                    ("read_file", "list_directory", ...)

 summarize_source stays a local Strands @tool
 (nested LLM call) — MCP replaces only the two
 "external world" tools, not the summarization step.

 Inspection path (separate, manual step):
 npx @modelcontextprotocol/inspector \
     npx @modelcontextprotocol/server-fetch
        │
        ▼
 browser UI at localhost:6274 → send tools/call →
 copy the request/response JSON-RPC pairs →
 mcp_inspector_trace.json
  • Strands' MCPClient spawns each MCP server as a child process and speaks JSON-RPC 2.0 over its stdin/stdout — this is the STDIO transport, the simplest and most common for local dev
  • Tools discovered from an MCP server (via the tools/list RPC call) are converted into Strands-compatible tool objects automatically — you don't write @tool wrappers for them yourself
  • You are consuming MCP servers here, not building one — that's a later-phase project

Prerequisites

  • Completed Project 1B (strands_version/research_agent.py works against Bedrock)
  • Node.js 18+ and npx available on PATH (the official MCP reference servers are npm packages, run via npx)
  • pip install strands-agents strands-agents-tools mcp (the mcp package provides the Python SDK types Strands' MCPClient builds on)
  • AWS credentials configured for Bedrock (same as 1B)
  • No install step needed for the MCP servers themselves — npx fetches and runs them on demand

Filesystem server needs an explicit allowed-directory argument. It refuses to serve paths outside whatever directories you pass on the command line — this is a deliberate sandboxing feature, not a bug. Never point it at your home directory for this exercise.

Step 2 — Connect Strands to the MCP servers via MCPClient

# mcp_version/research_agent.py
from mcp import StdioServerParameters, stdio_client
from strands import Agent, tool
from strands.models import BedrockModel
from strands.tools.mcp import MCPClient

fetch_mcp = MCPClient(
    lambda: stdio_client(
        StdioServerParameters(
            command="npx",
            args=["-y", "@modelcontextprotocol/server-fetch"],
        )
    )
)

filesystem_mcp = MCPClient(
    lambda: stdio_client(
        StdioServerParameters(
            command="npx",
            args=["-y", "@modelcontextprotocol/server-filesystem", "/tmp/mcp-sandbox"],
        )
    )
)

Each MCPClient lazily spawns its subprocess on first use inside a context manager. List and use the tools within that with block:

with fetch_mcp, filesystem_mcp:
    fetch_tools = fetch_mcp.list_tools_sync()
    fs_tools = filesystem_mcp.list_tools_sync()

    print("Fetch MCP tools:", [t.tool_name for t in fetch_tools])
    print("Filesystem MCP tools:", [t.tool_name for t in fs_tools])

Expect something like Fetch MCP tools: ['fetch'] and Filesystem MCP tools: ['read_file', 'write_file', 'list_directory', 'search_files', ...].

Step 3 — Replace web_search / fetch_page with MCP tools

The Fetch MCP server's fetch tool covers page retrieval (with built-in HTML-to-markdown extraction, replacing your httpx+trafilatura code entirely). It has no equivalent to a search-engine query tool, so keep your own web_search as a local Strands tool — MCP replaces what it has a real server for, nothing is forced:

from duckduckgo_search import DDGS


@tool
def web_search(query: str, max_results: int = 5) -> list[dict]:
    """Search the web for a query. Returns a list of dicts with
    'title', 'href', and 'body' keys, ranked by relevance."""
    with DDGS() as ddgs:
        return list(ddgs.text(query, max_results=max_results))


_summarizer = Agent(
    model=BedrockModel(model_id="us.anthropic.claude-haiku-4-5-20251001-v1:0"),
    system_prompt="Condense the given document into a factual 3-5 sentence summary.",
)


@tool
def summarize_source(text: str) -> str:
    """Summarize a fetched document's text into a short factual digest."""
    result = _summarizer(text[:12000])
    return str(result)


def build_agent(mcp_tools: list) -> Agent:
    model = BedrockModel(model_id="us.anthropic.claude-sonnet-4-5-20251001-v1:0")
    return Agent(
        model=model,
        system_prompt=(
            "You are a research assistant. Use web_search to find sources, "
            "the MCP 'fetch' tool to retrieve page content for the top 3 "
            "results, summarize_source on each, then synthesize a JSON "
            "ResearchBrief with fields: summary, key_findings, sources, "
            "confidence."
        ),
        tools=[web_search, summarize_source, *mcp_tools],
    )


def main() -> None:
    import sys

    topic = sys.argv[1] if len(sys.argv) > 1 else "quantum error correction 2026"

    with fetch_mcp:
        mcp_tools = fetch_mcp.list_tools_sync()
        agent = build_agent(mcp_tools)
        result = agent(topic)
        print(result)


if __name__ == "__main__":
    main()
What changed vs 1B: fetch_page (your ~15 lines of httpx/trafilatura code) is gone entirely — the Fetch MCP server's fetch tool is discovered at runtime via list_tools_sync() and dropped straight into the tools=[...] list alongside your local tools. Strands treats MCP-sourced and locally-defined tools identically once inside the agent loop.

Step 4 — Wire in the filesystem server for saved research notes

Give the agent a place to persist intermediate findings — a realistic use for the filesystem MCP server in a research pipeline:

with fetch_mcp, filesystem_mcp:
    mcp_tools = fetch_mcp.list_tools_sync() + filesystem_mcp.list_tools_sync()
    agent = build_agent(mcp_tools)
    agent.tool_registry.register_tool(summarize_source)  # if not passed at init

    result = agent(
        f"{topic}\n\nAfter synthesizing your findings, write the final "
        f"JSON ResearchBrief to a file named 'brief.json' using the "
        f"filesystem write_file tool."
    )
    print(result)
  • Verify the write actually happened: cat /tmp/mcp-sandbox/brief.json
  • This demonstrates composing two independent MCP servers in one agent — a pattern you'll reuse constantly once you're integrating third-party MCP servers you didn't write

Step 5 — Inspect the JSON-RPC exchange with MCP Inspector

npx @modelcontextprotocol/inspector npx @modelcontextprotocol/server-fetch
  • This opens a local web UI (default http://localhost:6274) that sits between you and the spawned server
  • In the UI: click Connect, then List Tools — you'll see the tools/list request/response pair rendered as JSON-RPC 2.0
  • Select the fetch tool, supply {"url": "https://example.com"} as arguments, and click Call Tool — this fires a tools/call request and shows the raw response
  • Inspector exposes a "History" or network-style panel showing every message; copy the full session as JSON (or use the browser DevTools Network tab on the Inspector's own backend calls if the UI doesn't export directly)

Save the captured exchange in this shape:

Step 6 — OAuth for remote/authenticated MCP servers (when you need it)

  • Everything above uses STDIO — a local subprocess you trust implicitly, so no auth layer is needed
  • The moment you connect to a remote, hosted MCP server (e.g. a SaaS product's MCP endpoint over HTTP/SSE) that guards access to private data, MCP v1.0 (Nov 2025) mandates the browser-based OAuth 2.1 flow: the client opens a browser window, the user authenticates and consents on the server's own login page, and a short-lived token is returned to the client — no credentials are ever typed into or stored by the MCP client itself
  • Strands' MCPClient supports this by pointing it at an HTTP/SSE transport instead of STDIO; the OAuth handshake is handled by the underlying mcp Python SDK's HTTP client, which will open a browser tab automatically on first connection to an auth-protected server
  • You do not need to implement this for the two local servers in this project — mention it in your README as the mechanism you'd reach for the moment you swap Fetch MCP for something like a hosted Brave Search MCP or a private company MCP server
  • Read the spec before you need it: MCP Authorization spec ↗

Troubleshooting

MCPClient hangs on startup: almost always means the subprocess command failed silently (wrong package name, npx couldn't resolve it, or Node isn't on PATH inside the environment Python is running in). Run the exact npx ... command standalone in the same shell first to confirm it starts.
"Tool not found" errors from the agent: the MCP client's with block context manager must still be open (subprocess alive) at the moment the agent actually calls the tool — don't call list_tools_sync() outside the with block and then exit it before running the agent.
Filesystem server refuses to read/write a path: it only serves the directories passed as CLI args at startup. Double-check the path in your prompt matches exactly (including trailing slashes) what you passed to server-filesystem.
MCP Inspector shows a connection error: some npm-hosted MCP servers pin specific Node versions; run node --version (need 18+) and retry with npx -y <package>@latest to rule out a stale cached version.
