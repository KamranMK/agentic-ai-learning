Build Guide: Multi-Agent Incident Response

Project 10 · Strands + LangGraph + A2A + AgentCore Memory/Observability/Identity · Reference: madhurprash/A2A-Multi-Agents-AgentCore ↗

What You'll Build

A simulated "website down" incident-response pipeline with three agents that mirrors a real on-call workflow: detect, diagnose, and remediate — with a human approval gate before anything destructive happens. This is the capstone-style project for Phase 2 because it exercises every AgentCore pillar (Runtime, Memory, Observability, Identity) plus A2A across a 3-hop chain.

  • Mock health-check service — a small FastAPI app with a /health endpoint and a toggle endpoint to force it to return 500s, simulating an outage.
  • Monitoring Agent (Strands) — polls /health on an interval, detects failures, and emits a structured IncidentReport Pydantic model to the Orchestrator over A2A.
  • Orchestrator Agent (Strands) — receives the incident, tracks the incident timeline in AgentCore Memory, and delegates root-cause diagnosis to the Remediation Agent over A2A.
  • Remediation Agent (LangGraph) — runs mocked runbook tools (check_logs(), check_recent_deploys(), restart_service()), with a hard human-in-the-loop interrupt before restart_service() ever executes.
  • All three agents deployed to separate AgentCore Runtime instances, secured with Amazon Cognito OAuth per the reference architecture.
  • An OTel trace of the full incident (detection → diagnosis → human approval → remediation) visible in AgentCore Observability / CloudWatch GenAI Observability.

Prerequisites

  • Completed Project 9 (two-agent A2A handoff) — you should already be comfortable with Agent Cards, JSON-RPC message/send, and agentcore launch.
  • AWS account with Bedrock model access, AgentCore Runtime, AgentCore Memory, AgentCore Observability, and Amazon Cognito enabled in your target region.
  • Python 3.11+, Docker, and the AgentCore CLI: pip install bedrock-agentcore bedrock-agentcore-starter-toolkit.
  • pip install strands-agents strands-agents-tools langgraph langchain-core pydantic a2a-sdk fastapi uvicorn boto3
  • Clone the community reference for a working end-to-end skeleton: git clone https://github.com/madhurprash/A2A-Multi-Agents-AgentCore.git — use it to sanity-check your Cognito setup, AgentCore Memory wiring, and multi-agent A2A folder layout before diverging into this guide's incident-response scenario.
  • An AWS IAM role with bedrock-agentcore:*, cognito-idp:* (for setup), and CloudWatch permissions for the observability trace.
Why fork the reference repo: Cognito OAuth setup for AgentCore (user pool, resource server, app client, scopes) is fiddly boilerplate that's easy to get wrong the first time. The reference repo has a working setup_cognito.py/CDK-style script — reuse its structure rather than reinventing it from the AWS docs cold.

Step 1 — Stand up the mock health-check service

# health_service/app.py
from fastapi import FastAPI, Response
import time

app = FastAPI()
_state = {"broken": False, "last_deploy": time.time() - 3600}

@app.get("/health")
def health():
    if _state["broken"]:
        return Response(status_code=500, content="internal server error")
    return {"status": "ok"}

@app.post("/toggle")
def toggle(broken: bool):
    _state["broken"] = broken
    return {"broken": _state["broken"]}

@app.get("/_meta/last_deploy")
def last_deploy():
    return {"last_deploy_epoch": _state["last_deploy"]}
uvicorn health_service.app:app --port 8000
# In another terminal, simulate an outage:
curl -X POST "http://localhost:8000/toggle?broken=true"

Step 2 — Define the shared IncidentReport model

# shared/models.py
from pydantic import BaseModel
from typing import Optional
from datetime import datetime

class IncidentReport(BaseModel):
    incident_id: str
    service_name: str
    detected_at: datetime
    endpoint: str
    consecutive_failures: int
    last_status_code: int
    severity: str  # "sev1" | "sev2" | "sev3"

class DiagnosisResult(BaseModel):
    incident_id: str
    root_cause_hypothesis: str
    recent_deploy_suspected: bool
    recommended_action: str
    logs_excerpt: str

class RemediationOutcome(BaseModel):
    incident_id: str
    action_taken: str
    approved_by_human: bool
    success: bool
    notes: str

Step 3 — Monitoring Agent (Strands)

# monitoring_agent/tools.py
from strands import tool
import httpx, time

_failure_streak = {"count": 0}

@tool
def poll_health(url: str = "http://localhost:8000/health") -> dict:
    """Poll the health-check endpoint once. Returns status_code and whether
    this extends a streak of consecutive failures (3+ triggers an incident)."""
    try:
        resp = httpx.get(url, timeout=3.0)
        code = resp.status_code
    except httpx.RequestError:
        code = 599
    if code >= 500:
        _failure_streak["count"] += 1
    else:
        _failure_streak["count"] = 0
    return {"status_code": code, "consecutive_failures": _failure_streak["count"]}
# monitoring_agent/agent.py
from strands import Agent
from strands.models import BedrockModel
from strands_tools.a2a_client import A2AClientToolProvider
from monitoring_agent.tools import poll_health
from shared.models import IncidentReport
import time, uuid

a2a_tools = A2AClientToolProvider(known_agent_urls=["http://localhost:9001/"])
model = BedrockModel(model_id="us.anthropic.claude-sonnet-4-5-20251001-v1:0")

monitor = Agent(
    model=model,
    system_prompt=(
        "You are a monitoring agent. Call poll_health repeatedly (the caller "
        "loop handles the interval). If consecutive_failures reaches 3, build "
        "an IncidentReport JSON object and call the Orchestrator agent's "
        "report_incident skill with it."
    ),
    tools=[poll_health] + a2a_tools.tools,
)

def run_loop(interval_s: int = 5):
    while True:
        result = monitor("Poll the health endpoint and report an incident if 3+ consecutive failures.")
        print(result)
        time.sleep(interval_s)

if __name__ == "__main__":
    run_loop()

Step 4 — Orchestrator Agent (Strands) with AgentCore Memory

The Orchestrator's Agent Card advertises a report_incident skill. On receiving a report, it writes the event to AgentCore Memory (so the timeline survives across the whole multi-agent conversation) and then hands diagnosis off to the Remediation Agent.

# orchestrator/memory.py
from bedrock_agentcore.memory import MemoryClient

memory_client = MemoryClient(region_name="us-east-1")
MEMORY_ID = "incident-response-memory"  # created once via memory_client.create_memory(...)

def record_event(session_id: str, actor_id: str, text: str):
    memory_client.create_event(
        memory_id=MEMORY_ID,
        actor_id=actor_id,
        session_id=session_id,
        messages=[(text, "ASSISTANT")],
    )

def get_timeline(session_id: str, actor_id: str):
    return memory_client.list_events(
        memory_id=MEMORY_ID, actor_id=actor_id, session_id=session_id
    )
# orchestrator/agent.py
from strands import Agent, tool
from strands.models import BedrockModel
from strands_tools.a2a_client import A2AClientToolProvider
from orchestrator.memory import record_event
from shared.models import IncidentReport

a2a_tools = A2AClientToolProvider(known_agent_urls=["http://localhost:9002/"])  # Remediation Agent
model = BedrockModel(model_id="us.anthropic.claude-sonnet-4-5-20251001-v1:0")

@tool
def log_incident(incident_json: str, session_id: str) -> str:
    """Persist an incoming IncidentReport to the AgentCore Memory timeline."""
    report = IncidentReport.model_validate_json(incident_json)
    record_event(session_id, "monitoring-agent", f"Incident {report.incident_id} detected: "
                 f"{report.consecutive_failures} consecutive {report.last_status_code} on {report.endpoint}")
    return f"logged incident {report.incident_id}"

orchestrator = Agent(
    model=model,
    system_prompt=(
        "You are the incident orchestrator. When you receive an IncidentReport, "
        "call log_incident to persist it, then call the Remediation Agent's "
        "diagnose_and_remediate skill with the same incident JSON. Relay its "
        "final RemediationOutcome back as your response."
    ),
    tools=[log_incident] + a2a_tools.tools,
)

Expose this as an A2A server the same way as Project 9 — an AgentExecutor plus an AgentCard with a report_incident skill, served on port 9001.

Step 5 — Remediation Agent (LangGraph) with a human-in-the-loop interrupt

This is the critical piece. Mock the three runbook tools, then use LangGraph's interrupt() to pause the graph before restart_service() fires. The graph must be compiled with a checkpointer so it can actually pause and resume — a plain in-memory graph without a checkpointer cannot survive an interrupt across process boundaries.

# remediation_agent/tools.py
def check_logs(service_name: str) -> str:
    """Mock: return a canned but realistic log excerpt."""
    return (
        f"[{service_name}] 03:14:02 ERROR OOMKilled: container exceeded memory limit\n"
        f"[{service_name}] 03:14:01 WARN  heap usage at 98% for 4m\n"
        f"[{service_name}] 03:12:55 INFO  request rate spike: 4200 req/min (baseline 800)"
    )

def check_recent_deploys(service_name: str) -> dict:
    """Mock: return the most recent deploy metadata."""
    return {
        "service": service_name,
        "last_deploy_minutes_ago": 18,
        "commit": "a1b2c3d",
        "author": "jsmith",
        "changed_files": ["cache_config.py", "worker_pool.py"],
    }

def restart_service(service_name: str) -> dict:
    """Mock: actually 'restarts' the service (flips the health toggle back)."""
    import httpx
    httpx.post("http://localhost:8000/toggle", params={"broken": False})
    return {"service": service_name, "restarted": True}
# remediation_agent/graph.py
from typing import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from langgraph.types import interrupt, Command
from remediation_agent.tools import check_logs, check_recent_deploys, restart_service
from shared.models import DiagnosisResult, RemediationOutcome

class RemediationState(TypedDict):
    incident_id: str
    service_name: str
    logs_excerpt: str
    deploy_info: dict
    diagnosis: DiagnosisResult
    human_decision: str
    outcome: RemediationOutcome

def gather_logs(state: RemediationState) -> dict:
    return {"logs_excerpt": check_logs(state["service_name"])}

def gather_deploys(state: RemediationState) -> dict:
    return {"deploy_info": check_recent_deploys(state["service_name"])}

def diagnose(state: RemediationState) -> dict:
    deploy_recent = state["deploy_info"]["last_deploy_minutes_ago"] < 30
    diagnosis = DiagnosisResult(
        incident_id=state["incident_id"],
        root_cause_hypothesis="Likely OOM caused by a memory regression in the recent deploy"
        if deploy_recent else "Likely traffic spike unrelated to recent deploys",
        recent_deploy_suspected=deploy_recent,
        recommended_action="restart_service",
        logs_excerpt=state["logs_excerpt"],
    )
    return {"diagnosis": diagnosis}

def human_approval_gate(state: RemediationState) -> dict:
    # Pauses graph execution here. The value passed to interrupt() is
    # surfaced to whatever is resuming the graph (e.g. a CLI prompt).
    decision = interrupt({
        "question": f"Approve restart_service for '{state['service_name']}'? "
                     f"Root cause: {state['diagnosis'].root_cause_hypothesis}",
        "incident_id": state["incident_id"],
    })
    return {"human_decision": decision}

def apply_remediation(state: RemediationState) -> dict:
    if state["human_decision"] != "approve":
        outcome = RemediationOutcome(
            incident_id=state["incident_id"], action_taken="none",
            approved_by_human=False, success=False,
            notes="Human rejected the proposed restart.",
        )
        return {"outcome": outcome}
    result = restart_service(state["service_name"])
    outcome = RemediationOutcome(
        incident_id=state["incident_id"], action_taken="restart_service",
        approved_by_human=True, success=result["restarted"],
        notes="Service restarted after human approval.",
    )
    return {"outcome": outcome}

graph = StateGraph(RemediationState)
graph.add_node("gather_logs", gather_logs)
graph.add_node("gather_deploys", gather_deploys)
graph.add_node("diagnose", diagnose)
graph.add_node("human_approval_gate", human_approval_gate)
graph.add_node("apply_remediation", apply_remediation)
graph.add_edge(START, "gather_logs")
graph.add_edge("gather_logs", "gather_deploys")
graph.add_edge("gather_deploys", "diagnose")
graph.add_edge("diagnose", "human_approval_gate")
graph.add_edge("human_approval_gate", "apply_remediation")
graph.add_edge("apply_remediation", END)

checkpointer = MemorySaver()
remediation_graph = graph.compile(checkpointer=checkpointer)

Driving the interrupt/resume cycle from the agent executor — this is where the terminal approval prompt lives:

# remediation_agent/agent_executor.py
from a2a.server.agent_execution import AgentExecutor, RequestContext
from a2a.server.events import EventQueue
from a2a.utils import new_task, new_agent_text_message
from remediation_agent.graph import remediation_graph
from langgraph.types import Command
import json

class RemediationExecutor(AgentExecutor):
    async def execute(self, context: RequestContext, event_queue: EventQueue) -> None:
        task = context.current_task or new_task(context.message)
        await event_queue.enqueue_event(task)

        incident = json.loads(context.get_user_input())
        thread_config = {"configurable": {"thread_id": incident["incident_id"]}}

        result = remediation_graph.invoke(
            {"incident_id": incident["incident_id"], "service_name": incident.get("service_name", "web-frontend")},
            config=thread_config,
        )

        if "__interrupt__" in result:
            interrupt_payload = result["__interrupt__"][0].value
            print(f"\n[HUMAN APPROVAL REQUIRED] {interrupt_payload['question']}")
            decision = input("Type 'approve' or 'deny': ").strip().lower()
            result = remediation_graph.invoke(Command(resume=decision), config=thread_config)

        outcome = result["outcome"]
        await event_queue.enqueue_event(
            new_agent_text_message(outcome.model_dump_json(), task.context_id, task.id)
        )
Why this matters: The interrupt is not a UI nicety — it's the load-bearing safety control of the whole project. restart_service() only executes on the branch reached after Command(resume="approve") re-enters the graph past human_approval_gate. If you skip the checkpointer, interrupt() has nothing to resume into and the graph will error or silently re-run from the start.

Step 6 — Secure all three runtimes with Cognito OAuth

Follow the reference architecture's pattern: one Cognito User Pool, one resource server per agent (or a shared resource server with per-agent scopes), and an app client used by callers (Monitoring → Orchestrator → Remediation) to obtain bearer tokens via client-credentials flow.

\# Create user pool + resource server + app client (adapt from the reference repo's setup script)
aws cognito-idp create-user-pool --pool-name incident-response-pool
aws cognito-idp create-resource-server \
  --user-pool-id  \
  --identifier incident-agents \
  --name "Incident Response Agents" \
  --scopes ScopeName=invoke,ScopeDescription="Invoke agent runtime"
aws cognito-idp create-user-pool-client \
  --user-pool-id  \
  --client-name agent-caller \
  --generate-secret \
  --allowed-o-auth-flows client_credentials \
  --allowed-o-auth-scopes incident-agents/invoke \
  --allowed-o-auth-flows-user-pool-client
# agentcore configure --authorizer-config to require the Cognito JWT on invocation
agentcore configure \
  --entrypoint agent.py \
  --name orchestrator \
  --authorizer-config '{"customJWTAuthorizer": {"discoveryUrl": "https://cognito-idp.us-east-1.amazonaws.com//.well-known/openid-configuration", "allowedClients": [""]}}'

Each agent's A2A client (Monitoring calling Orchestrator, Orchestrator calling Remediation) must first fetch a token from Cognito's token endpoint and attach it as Authorization: Bearer <token> on the outbound JSON-RPC POST.

import httpx

def get_cognito_token(client_id, client_secret, token_url, scope):
    resp = httpx.post(token_url, data={
        "grant_type": "client_credentials",
        "client_id": client_id,
        "client_secret": client_secret,
        "scope": scope,
    })
    return resp.json()["access_token"]

Each A2A hop (Monitoring → Orchestrator, Orchestrator → Remediation), each LangGraph node execution, and the interrupt/resume boundary should appear as distinct spans in the trace, correlated by a shared trace/session ID.

Verification

  • Toggle the outage on: curl -X POST "http://localhost:8000/toggle?broken=true", then run the Monitoring Agent loop and confirm it detects 3 consecutive failures and fires an A2A call to the Orchestrator within ~3 poll intervals.
  • Confirm the Orchestrator wrote the timeline entry: memory_client.list_events(memory_id=MEMORY_ID, actor_id="monitoring-agent", session_id=<id>) returns the logged incident text.
  • Confirm the Remediation Agent actually pauses: watch for [HUMAN APPROVAL REQUIRED] ... printed to the terminal before restart_service() ever runs — grep your logs to prove restart_service was NOT called until after you typed approve.
  • Type deny once in a separate run and confirm RemediationOutcome.success == false and the health endpoint remains broken (curl http://localhost:8000/health still 500).
  • Type approve and confirm curl http://localhost:8000/health now returns {"status": "ok"}.
  • Open CloudWatch GenAI Observability (or run agentcore observability trace --session-id <id> if using the CLI viewer) and confirm you can see a single trace spanning all three agents with the interrupt as a visible gap in the span timeline.

Troubleshooting

  • Graph resumes from the start instead of after the interrupt: you're invoking without the same thread_id in config, or you swapped MemorySaver() for a fresh instance between the initial call and the resume call. The checkpointer instance and thread_id must persist across both invocations of remediation_graph.invoke(...).
  • Cognito 401s on inter-agent calls: the token's scope claim must match what the receiving AgentCore Runtime's customJWTAuthorizer.allowedClients/scope config expects — decode the JWT (jwt.io or python -c "import jwt; print(jwt.decode(token, options={'verify_signature': False}))") and confirm scope and client_id match your resource server config exactly.
  • Monitoring Agent never fires an incident: confirm the toggle actually persisted (state is process-local to the FastAPI app — restarting uvicorn resets it to healthy) and that consecutive_failures resets to 0 on any single 200 in between polls, which can happen if you're running against a load balancer with multiple healthy replicas behind it.
  • AgentCore Runtime port conflicts locally: each agent needs a distinct local port when testing with agentcore launch --local before deploying — reuse the 9000/9001/9002 convention from this guide and double check nothing else on your machine is already bound to those ports (lsof -i :9001).
  • Observability trace is missing the interrupt gap: confirm you launched with --enable-observability and that both the pre-interrupt and post-resume invoke() calls share the same session/trace correlation ID passed through the A2A message metadata — otherwise CloudWatch shows two disconnected traces instead of one.
