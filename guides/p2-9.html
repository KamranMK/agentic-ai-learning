Build Guide: Two-Agent A2A Handoff

Project 9 · Strands + LangGraph + Agent2Agent (A2A) protocol · AWS Bedrock AgentCore

What You'll Build

A trip-planning system split across two independently deployable agents that talk to each other over the open A2A protocol instead of in-process function calls. This is the first project where "the other agent" is a separate service you discover at runtime via a published Agent Card, not a Python object you import.

  • Concierge agent — built with Strands SDK + Claude on Bedrock. Takes a natural-language trip request (e.g. "3-day trip to Lisbon, $1500 budget, loves food and architecture"), extracts structured constraints, and delegates itinerary generation to a remote agent over A2A.
  • Itinerary Builder agent — built with LangGraph. Receives the delegated task, runs a small graph that produces a day-by-day plan, and returns it as structured JSON validated against a Pydantic model.
  • Two A2A Agent Cards (JSON) — one per agent, each advertising its skills per the A2A spec.
  • A working JSON-RPC 2.0 handoff: Concierge discovers Itinerary Builder's card, sends a message/send request, and receives a structured task result back.
  • Both agents running locally on AgentCore Runtime (ports 9000/9001), then deployed with agentcore launch as two separate AgentCore Runtime resources.

Architecture

┌─────────────────────────┐            ┌──────────────────────────────┐
│   Concierge Agent        │            │   Itinerary Builder Agent      │
│   (Strands + Bedrock)    │            │   (LangGraph)                  │
│   AgentCore Runtime      │            │   AgentCore Runtime             │
│   :9000                  │            │   :9001                        │
│                          │            │                                │
│  1. Parse user request  │            │  4. Receive JSON-RPC task      │
│  2. GET /.well-known/    │───────────▶│     via /a2a/message/send      │
│     agent-card.json      │  discover  │  5. Run LangGraph:              │
│     from Itinerary       │            │     plan_days → assign_        │
│     Builder               │            │     activities → estimate_     │
│  3. POST JSON-RPC 2.0    │            │     costs → format_itinerary   │
│     message/send with    │───────────▶│  6. Validate output against    │
│     trip constraints     │  A2A call  │     Itinerary pydantic model   │
│                          │◀───────────│  7. Return JSON-RPC result     │
│  8. Render final reply   │  response  │     (day-by-day plan)          │
│     to the user           │            │                                │
└─────────────────────────┘            └──────────────────────────────┘

The key architectural shift from Phase 1: Concierge never imports Itinerary Builder's code. It only knows a URL. Everything about capability discovery (what skills does this agent have?) and invocation (how do I call it, what does it expect?) is described by the Agent Card and the A2A JSON-RPC transport — the same pattern you'd use if Itinerary Builder were built and owned by a completely different team.

Prerequisites

  • Completed Phase 1 — comfortable writing a Strands agent and calling Bedrock models; basic PydanticAI/Pydantic model familiarity.
  • AWS account with Bedrock model access enabled for Claude (Anthropic models) in your region, plus IAM permissions for Bedrock AgentCore.
  • Python 3.11+
  • pip install strands-agents strands-agents-tools langgraph langchain-core pydantic a2a-sdk bedrock-agentcore bedrock-agentcore-starter-toolkit
  • AgentCore CLI (agentcore) installed and configured — pip install bedrock-agentcore-starter-toolkit ships the agentcore command.
  • AWS credentials configured (aws configure or environment variables) with permission to create AgentCore Runtime resources.
  • Docker (AgentCore builds a container image for each runtime on agentcore launch).
Note: The A2A protocol is transport-agnostic JSON-RPC 2.0 over HTTP(S). You don't need a special AWS-only library to speak it — a2a-sdk (the reference Python implementation) works the same whether the peer agent runs on AgentCore, ECS, or your laptop.

Step 1 — Define the shared itinerary schema

Both agents need to agree on the shape of the final itinerary. Put this in a small shared module (or just duplicate it — the contract is the JSON, not the Python class).

# shared/models.py
from pydantic import BaseModel, Field
from typing import List

class Activity(BaseModel):
    time_of_day: str = Field(..., description="morning | afternoon | evening")
    title: str
    description: str
    estimated_cost_usd: float

class DayPlan(BaseModel):
    day_number: int
    theme: str
    activities: List[Activity]
    day_total_usd: float

class Itinerary(BaseModel):
    destination: str
    trip_days: int
    budget_usd: float
    interests: List[str]
    days: List[DayPlan]
    total_estimated_cost_usd: float

Step 2 — Build the Itinerary Builder agent (LangGraph)

A small linear graph: plan the day structure, assign activities per day, estimate costs, then format into the Itinerary model.

# itinerary_builder/graph.py
from typing import TypedDict, List
from langgraph.graph import StateGraph, START, END
from langchain_core.messages import HumanMessage
from langchain_anthropic import ChatAnthropic
from shared.models import Itinerary, DayPlan, Activity

class BuilderState(TypedDict):
    destination: str
    trip_days: int
    budget_usd: float
    interests: List[str]
    days: List[DayPlan]
    itinerary: Itinerary

llm = ChatAnthropic(model="claude-sonnet-4-5-20251001", temperature=0.4)

def plan_days(state: BuilderState) -> dict:
    themes = [f"Day {i+1} focus" for i in range(state["trip_days"])]
    return {"days": []}

def assign_activities(state: BuilderState) -> dict:
    per_day_budget = state["budget_usd"] / state["trip_days"]
    days = []
    for i in range(state["trip_days"]):
        prompt = (
            f"Suggest 3 activities for day {i+1} of a trip to {state['destination']} "
            f"for someone who loves {', '.join(state['interests'])}. "
            f"Budget for the day is roughly ${per_day_budget:.0f}. "
            "Return a short JSON list of {title, description, time_of_day, estimated_cost_usd}."
        )
        resp = llm.invoke([HumanMessage(content=prompt)])
        activities = _parse_activities(resp.content)  # your JSON-extraction helper
        day_total = sum(a.estimated_cost_usd for a in activities)
        days.append(DayPlan(day_number=i + 1, theme=f"Day {i+1}",
                             activities=activities, day_total_usd=day_total))
    return {"days": days}

def estimate_costs(state: BuilderState) -> dict:
    total = sum(d.day_total_usd for d in state["days"])
    return {"_total": total}

def format_itinerary(state: BuilderState) -> dict:
    total = sum(d.day_total_usd for d in state["days"])
    itinerary = Itinerary(
        destination=state["destination"], trip_days=state["trip_days"],
        budget_usd=state["budget_usd"], interests=state["interests"],
        days=state["days"], total_estimated_cost_usd=total,
    )
    return {"itinerary": itinerary}

graph = StateGraph(BuilderState)
graph.add_node("plan_days", plan_days)
graph.add_node("assign_activities", assign_activities)
graph.add_node("estimate_costs", estimate_costs)
graph.add_node("format_itinerary", format_itinerary)
graph.add_edge(START, "plan_days")
graph.add_edge("plan_days", "assign_activities")
graph.add_edge("assign_activities", "estimate_costs")
graph.add_edge("estimate_costs", "format_itinerary")
graph.add_edge("format_itinerary", END)
itinerary_graph = graph.compile()

Step 3 — Wrap the LangGraph app as an A2A server

The A2A Python SDK (a2a-sdk) gives you an AgentExecutor interface: implement execute() to run your logic and push results onto the event queue. This is what makes Itinerary Builder speak JSON-RPC 2.0 over HTTP without you hand-rolling the protocol.

# itinerary_builder/agent_executor.py
from a2a.server.agent_execution import AgentExecutor, RequestContext
from a2a.server.events import EventQueue
from a2a.types import Task, TaskState, TextPart
from a2a.utils import new_agent_text_message, new_task
from itinerary_builder.graph import itinerary_graph
import json

class ItineraryBuilderExecutor(AgentExecutor):
    async def execute(self, context: RequestContext, event_queue: EventQueue) -> None:
        task = context.current_task or new_task(context.message)
        await event_queue.enqueue_event(task)

        params = json.loads(context.get_user_input())  # {destination, trip_days, budget_usd, interests}
        result_state = itinerary_graph.invoke(params)
        itinerary = result_state["itinerary"]

        await event_queue.enqueue_event(
            new_agent_text_message(itinerary.model_dump_json(), task.context_id, task.id)
        )

    async def cancel(self, context: RequestContext, event_queue: EventQueue) -> None:
        raise NotImplementedError("Itinerary Builder does not support cancellation")
# itinerary_builder/server.py
from a2a.server.apps import A2AStarletteApplication
from a2a.server.request_handlers import DefaultRequestHandler
from a2a.server.tasks import InMemoryTaskStore
from a2a.types import AgentCard, AgentCapabilities, AgentSkill
from itinerary_builder.agent_executor import ItineraryBuilderExecutor
import uvicorn

skill = AgentSkill(
    id="build_itinerary",
    name="Build Day-by-Day Itinerary",
    description="Generates a structured, budget-aware day-by-day travel itinerary "
                "given a destination, trip length, budget, and traveler interests.",
    tags=["travel", "itinerary", "planning"],
    examples=["3-day trip to Lisbon, $1500 budget, loves food and architecture"],
)

agent_card = AgentCard(
    name="Itinerary Builder",
    description="Generates structured multi-day travel itineraries with cost estimates.",
    url="http://localhost:9001/",
    version="1.0.0",
    default_input_modes=["text/plain"],
    default_output_modes=["application/json"],
    capabilities=AgentCapabilities(streaming=False),
    skills=[skill],
)

handler = DefaultRequestHandler(
    agent_executor=ItineraryBuilderExecutor(),
    task_store=InMemoryTaskStore(),
)
app = A2AStarletteApplication(agent_card=agent_card, http_handler=handler)

if __name__ == "__main__":
    uvicorn.run(app.build(), host="0.0.0.0", port=9001)

This automatically serves the Agent Card at GET http://localhost:9001/.well-known/agent-card.json. Here's the full JSON that gets exposed:

  "url": "http://localhost:9001/",
  "version": "1.0.0",
  "protocolVersion": "0.3.0",
  "defaultInputModes": ["text/plain"],
  "defaultOutputModes": ["application/json"],
  "capabilities": { "streaming": false },
  "skills": [
    {
      "id": "build_itinerary",
      "name": "Build Day-by-Day Itinerary",
      "description": "Generates a structured, budget-aware day-by-day travel itinerary given a destination, trip length, budget, and traveler interests.",
      "tags": ["travel", "itinerary", "planning"],

    }
  ]
}

Step 4 — Build the Concierge agent (Strands) with A2A discovery

Strands ships an strands_tools.a2a_client tool (and the underlying A2AClientToolProvider) that fetches a remote Agent Card and exposes it as a callable tool. Concierge's system prompt does the natural-language parsing; the tool does the handoff.

# concierge/agent.py
from strands import Agent
from strands.models import BedrockModel
from strands_tools.a2a_client import A2AClientToolProvider

# Discover Itinerary Builder by its base URL — the provider fetches
# /.well-known/agent-card.json under the hood and turns each skill
# into a tool the LLM can call.
a2a_tools = A2AClientToolProvider(known_agent_urls=["http://localhost:9001/"])

model = BedrockModel(model_id="us.anthropic.claude-sonnet-4-5-20251001-v1:0")

concierge = Agent(
    model=model,
    system_prompt=(
        "You are a travel concierge. Extract destination, trip length in days, "
        "budget in USD, and interests from the user's request. Then call the "
        "Itinerary Builder agent's build_itinerary skill with those fields as "
        "a JSON object. Present the returned itinerary back to the user in a "
        "friendly, readable format."
    ),
    tools=a2a_tools.tools,
)

if __name__ == "__main__":
    response = concierge("3-day trip to Lisbon, $1500 budget, loves food and architecture")
    print(response)
What's happening under the hood: On agent construction, A2AClientToolProvider does GET http://localhost:9001/.well-known/agent-card.json, reads the skills array, and registers a Strands tool per skill. When the LLM decides to call it, the provider issues a JSON-RPC 2.0 message/send POST to the agent's url and returns the task result as the tool's output.

Step 5 — The JSON-RPC 2.0 handoff, concretely

When Concierge's LLM decides to call the build_itinerary skill, the A2A client sends this over HTTP POST to http://localhost:9001/:

{
  "jsonrpc": "2.0",
  "id": "req-001",
  "method": "message/send",
  "params": {
    "message": {
      "role": "user",
      "parts": [
        { "kind": "text", "text": "{\"destination\": \"Lisbon\", \"trip_days\": 3, \"budget_usd\": 1500, \"interests\": [\"food\", \"architecture\"]}" }
      ],
      "messageId": "msg-001"
    }
  }
}

Itinerary Builder responds with a Task object wrapping the generated itinerary as the artifact text:

{
  "jsonrpc": "2.0",
  "id": "req-001",
  "result": {
    "id": "task-9f21",
    "contextId": "ctx-7b3a",
    "status": { "state": "completed" },
    "artifacts": [
      {
        "artifactId": "art-1",
        "parts": [
          { "kind": "text", "text": "{\"destination\": \"Lisbon\", \"trip_days\": 3, \"budget_usd\": 1500.0, \"interests\": [\"food\", \"architecture\"], \"days\": [ { \"day_number\": 1, \"theme\": \"Day 1\", \"activities\": [ { \"time_of_day\": \"morning\", \"title\": \"Belém Tower & pastéis de nata\", \"description\": \"Walk the riverside district and try the original custard tart.\", \"estimated_cost_usd\": 25.0 } ], \"day_total_usd\": 25.0 } ], \"total_estimated_cost_usd\": 480.0 }" }
        ]
      }
    ]
  }
}

Concierge's tool wrapper extracts the artifact text, and the LLM parses/validates it against the shared Itinerary Pydantic model before presenting it to the user.

Step 6 — Run both agents locally

# terminal 1
python -m itinerary_builder.server        # serves on :9001

# terminal 2
python -m concierge.agent                  # Concierge calls out to :9001
Local dev port convention: The spec for this project uses port 9000 as the reserved default for whichever agent you're actively developing/debugging with agentcore launch --local; give the peer agent an adjacent port (9001) so both can run simultaneously without collision.

Step 7 — Deploy both agents to AgentCore Runtime

Each agent gets its own AgentCore Runtime deployment. Configure with agentcore configure, then launch.

# In itinerary_builder/ directory
agentcore configure --entrypoint server.py --name itinerary-builder
agentcore launch

# In concierge/ directory
agentcore configure --entrypoint agent.py --name concierge
agentcore launch

After both are deployed, update Concierge's known_agent_urls to point at Itinerary Builder's deployed AgentCore Runtime endpoint (returned by agentcore launch as the runtime ARN/invoke URL) instead of localhost:9001, then redeploy Concierge:

agentcore launch --update
# Invoke the deployed Concierge agent
agentcore invoke '{"prompt": "3-day trip to Lisbon, $1500 budget, loves food and architecture"}'

Verification

  • Fetch the Agent Card directly to confirm discovery works: curl http://localhost:9001/.well-known/agent-card.json | jq .skills — should list build_itinerary.
  • Run python -m concierge.agent and confirm the printed response contains a day-by-day breakdown with day_total_usd figures that sum to roughly the requested budget.
  • Set logging.getLogger("a2a").setLevel(logging.DEBUG) (and Strands' own DEBUG logger) before running Concierge to print the raw JSON-RPC request/response pairs to stdout — confirm the outbound method is message/send and the inbound status.state is completed.
  • Validate the returned itinerary JSON against the Itinerary Pydantic model explicitly in a test: Itinerary.model_validate_json(artifact_text) should not raise.
  • On AgentCore: agentcore status --name itinerary-builder and agentcore status --name concierge should both show READY; check CloudWatch Logs for the runtime's log group to see the JSON-RPC trace in production.

Troubleshooting

  • Agent Card discovery fails (404 or connection refused): confirm Itinerary Builder is actually running and bound to 0.0.0.0 (not 127.0.0.1) if Concierge is calling it from a different container/host. Locally, verify with curl -i http://localhost:9001/.well-known/agent-card.json before touching Concierge code.
  • Port conflicts on AgentCore Runtime: AgentCore Runtime expects your server to listen on the port declared at agentcore configure time (default 8080 in the container, regardless of your local dev port). If your app hardcodes 9001 inside the container, override the bind port via the PORT env var AgentCore injects rather than hardcoding it.
  • LLM never calls the A2A tool: the tool's description in the Agent Card's skills[].description is what Concierge's LLM reads to decide relevance — make it specific ("generates a day-by-day itinerary...") not generic ("does travel stuff").
  • Auth errors calling the deployed AgentCore endpoint: AgentCore Runtime invocations require SigV4-signed requests (or an inbound identity token if you've configured an auth provider) — confirm your local AWS credentials/role has bedrock-agentcore:InvokeAgentRuntime permission, and that Concierge's outbound call to Itinerary Builder's deployed URL includes proper credentials rather than the bare local-dev HTTP call.
