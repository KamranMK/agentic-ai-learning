Agentic AI Evolution & Future Framing

  • Spectrum: low-agency (rule-based, human oversight) → high-agency (autonomous, self-directed decisions)
  • We are still in the early stages of high-agency AI — reaching it requires not just tech but governance + organizational readiness
  • Sequoia Capital prediction: by the 2030s, AI agents will form a "global neural network" — interconnected intelligent agents
  • "One-person unicorn": a single developer with AI agents performing all functional capabilities of a full startup
  • MCP and A2A protocols = TCP/IP for agents — just as TCP/IP enabled the internet, these protocols enable the agent economy

Amazon Nova Act: Web Agent Capability

  • Nova Act = Amazon's browser-use model — can scrape web pages, navigate sites, extract data
  • Operates like a human web engineer: given a URL and instructions in natural language, it finds and extracts content
  • Exposed as an MCP tool so any MCP-compatible agent (Strands, Claude Desktop) can use it
from strands import Agent
from strands.tools.mcp import MCPClient
import threading

# Start Nova Act as an MCP server in background thread
def run_nova_act_server():
    # Nova Act runs as stdio MCP server
    subprocess.run(["nova-act", "--transport", "stdio"])

thread = threading.Thread(target=run_nova_act_server, daemon=True)
thread.start()

# Connect Strands agent to Nova Act via MCP
with MCPClient(["nova-act", "--transport", "stdio"]) as client:
    agent = Agent(
        system_prompt="You are a weather assistant for Hong Kong.",
        tools=client.list_tools()  # discovers forecast_weather, etc.
    )
    agent("What is the current 9-day weather forecast in Hong Kong?")
  • Nova Act reads the HTML page, extracts the forecast data, saves a screenshot for verification
  • Practical for enterprise automation: crawl internal dashboards, extract reports, monitor changes

Strands + Manim: Educational Video Agent (Demo)

  • Manim = Python library for mathematical animations (used by 3Blue1Brown)
  • Strands agent connects to a Manim MCP server → receives natural language math requests → generates animation videos
  • "Visualize the cubic function f(x) = x³ from x=-3 to x=3" → agent writes Manim code → renders MP4
  • When the agent encounters a dependency issue, it adapts and creates a simplified version — demonstrates adaptive problem solving
  • Entire implementation built in ~30 minutes with a few dozen lines of Python
Pattern: Any Python library can become an agent capability by wrapping it as an MCP server. This unlocks code generation, data visualization, report generation as agentic workflows.

Why Multi-Agent Architecture?

  • Specialization: each agent focuses on a domain → higher precision than a single jack-of-all-trades agent
  • Scalability: add or remove sub-agents without touching the orchestrator
  • Maintainability: debug 100 lines of a focused agent, not 10,000 lines of a monolithic one
  • Cost optimization: match task complexity to model — use fast/cheap models for classification, powerful models for reasoning
  • Analogy: single agent = solo trader managing everything. Multi-agent = hedge fund team (analyst, quant, trader, risk manager, IT)

Four Multi-Agent Collaboration Patterns

1 Agent Tools (Orchestrator + Sub-Agents)
When to use: clear delegation of specialized tasks; you know in advance what each sub-agent does
  • Orchestrator receives user request → delegates to specialized sub-agents as tool calls
  • Sub-agent output becomes input to the next sub-agent (pipeline)
  • AgentCore Runtime hosts sub-agents; orchestrator calls them via AgentCore Gateway
  • Sub-agents deployed with @app.entrypoint and invoked via agentcore_runtime.invoke()
from strands import Agent, tool
from bedrock_agentcore import BedrockAgentCoreApp

# Sub-agent: deploys to AgentCore Runtime
app = BedrockAgentCoreApp()

@app.entrypoint
def strategy_generator(payload):
    agent = Agent(
        model="claude-sonnet",  # good for code generation
        system_prompt="You are a trading strategy code generator. Use BackTrader framework."
    )
    return {"code": str(agent(payload["strategy_params"]))}

# Orchestrator: calls sub-agent as a tool
@tool
def generate_trading_strategy(name: str, buy_condition: str, sell_condition: str) -> str:
    """Generate a BackTrader strategy Python script from parameters."""
    response = agentcore_runtime.invoke(
        agent_id="strategy-generator",
        payload={"strategy_params": {"name": name, "buy": buy_condition, "sell": sell_condition}}
    )
    return response["code"]
Demo use case: Quant Research Agent — orchestrator calls: (1) strategy generator, (2) market data fetcher (via Gateway → DynamoDB), (3) back-test runner, (4) results summarizer. Each runs in AgentCore Runtime with its own model.
2 Swarm (Peer Collaboration, No Central Boss)
When to use: multi-disciplinary problems where no single agent owns the solution; emergent behavior needed
  • Agents communicate directly with each other; shared context (not a central orchestrator)
  • Any agent can hand off to any other agent based on what it determines is needed
  • Uses handoff detection to prevent two agents ping-ponging endlessly
from strands import Agent
from strands.multiagent import Swarm

research_agent = Agent(system_prompt="You are a researcher. Find facts and data.")
creative_agent = Agent(system_prompt="You are a writer. Create engaging content.")
critic_agent   = Agent(system_prompt="You are a critic. Challenge and improve the content.")
summary_agent  = Agent(system_prompt="You are a summarizer. Create final polished output.")

swarm = Swarm(
    agents=[research_agent, creative_agent, critic_agent, summary_agent],
    max_handoffs=10,
    repetitive_handoff_detection_window=8,  # last 8 handoffs must include ≥3 unique agents
    repetitive_handoff_min_unique_agents=3
)

result = swarm.run("Create a blog post explaining Agentic AI with a social media summary.")
Demo behavior: Research → Creative → Summary, but Summary says "I need more detail" and hands back to Creative → Creative fills content → Summary produces final post. 5 interactions total, emergent.
3 Graph (Controlled Information Flow)
When to use: predictability required; specific agents must communicate only with specific others; compliance workflows
  • Nodes = agents. Edges = allowed communication paths. You define who can talk to whom.
  • Key advantage over swarm: deterministic information routing — no agent talks to an unauthorized agent
  • Works with LangGraph and other graph frameworks; Strands integrates with LangGraph natively
from strands import Agent
from langgraph.graph import StateGraph

financial_agent  = Agent(system_prompt="You are a financial analyst.")
portfolio_agent  = Agent(system_prompt="You are a portfolio constructor.")
risk_agent       = Agent(system_prompt="You are a risk manager.")

# Define graph: financial → portfolio → risk (risk is terminal)
workflow = StateGraph(state_schema=dict)
workflow.add_node("financial",  lambda s: {"analysis": financial_agent(s["request"])})
workflow.add_node("portfolio",  lambda s: {"portfolio": portfolio_agent(s["analysis"])})
workflow.add_node("risk",       lambda s: {"verdict": risk_agent(s["portfolio"])})

workflow.add_edge("financial", "portfolio")
workflow.add_edge("portfolio", "risk")
workflow.set_entry_point("financial")

graph = workflow.compile()
result = graph.invoke({"request": "Construct a portfolio for moderate risk tolerance"})
Use case: Data validation pipeline — validator agent routes to either error-handling agent or success-processing agent based on validation result. Full control over which branch is taken.
4 Workflow (Sequential Steps with Dependencies)
When to use: complex repeatable processes with known steps; data pipelines; ETL; report generation
  • Tasks have explicit dependencies — task B cannot start until task A completes
  • Priority setting enables parallel execution where tasks don't depend on each other
from strands_tools import workflow, WorkflowTask

tasks = [
    WorkflowTask(id="extract",   task="Extract data from S3 sources"),
    WorkflowTask(id="transform", task="Clean and transform data", depends_on=["extract"]),
    WorkflowTask(id="analyze",   task="Run trend analysis",        depends_on=["transform"]),
    WorkflowTask(id="report",    task="Generate executive report", depends_on=["analyze"]),
]

from strands import Agent
from strands_tools import workflow as workflow_tool

agent = Agent(
    system_prompt="You run data pipeline workflows.",
    tools=[workflow_tool]
)
agent("Execute the quarterly analytics pipeline")
Perfect for: Automated data engineering pipelines — the exact capstone project (Autonomous DS Pipeline Agent) this plan builds toward.

Pattern Selection Guide

  • Agent Tools: customer service routing, quant research pipeline, any orchestrator-delegates workflow
  • Swarm: multi-disciplinary incident response (network → database → app agents), creative collaboration, situations where you want emergent coordination
  • Graph: validation pipelines with error branches, compliance workflows where you need to prove what talked to what
  • Workflow: data ETL, report generation, any complex repeatable process with known step dependencies

Key Takeaways

  • The four patterns (Agent Tools, Swarm, Graph, Workflow) cover nearly all multi-agent use cases — map your problem to one before building
  • Strands integrates with LangGraph, LangChain, and other frameworks — you don't have to choose one and commit forever
  • Use different models for different sub-agents: Claude Sonnet for code generation, Nova for summarization → cost efficiency
  • Nova Act = browser automation as an MCP tool — any web page becomes a data source for your agent
  • AgentCore Gateway separates auth concern from agent code — agents call tools, Gateway handles credentials
  • AgentCore Memory: pass context between agents without huge prompt payloads — store in short-term, retrieve semantically in long-term
