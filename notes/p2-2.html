Problem: Why AML Needs AI Agents

  • $3.1 trillion in illicit funds flowed globally in 2023 — drug/human trafficking, terrorism, fraud
  • Traditional rule-based systems produce up to 95% false positives — compliance staff spend most time on legitimate transactions
  • Diverse data: financial transactions, KYC profiles, news feeds, sanctions lists — hard to integrate
  • Agents solve this: context-aware reasoning across structured + unstructured data, with lower false alarm rates

When to Use Agentic Workflow vs. Traditional

  • Use agentic: complex workflows, diverse/unstructured data, memory needed across turns, multi-step reasoning required
  • Use traditional: mission-critical with 100% determinism required, ultra-low latency needed, fixed rule-based logic
Key insight: The AML solution combines both — classical ML risk models run inside the agentic workflow as one of the tools. Don't throw out your existing models; extend them with agents.

Technology Stack

  • Strands Agents SDK — builds and runs individual agents (a few lines of code per agent)
  • Amazon Bedrock — model provider (Claude for reasoning, Nova for classification)
  • AgentCore Runtime — serverless microVM for deploying agents; sessions up to 8 hours
  • AgentCore Gateway — secure proxy connecting agents to Lambda functions, API Gateway, MCP tools
  • AgentCore Memory — short-term (up to 365 days) + long-term with semantic extraction
  • Amazon Kinesis — real-time transaction ingestion stream
  • Amazon DynamoDB — historical transaction store (fast retrieval for last N days)
  • Amazon Bedrock Knowledge Base — vector store for sanctions lists and regulatory documents
  • OpenSearch Serverless — audit trail storage, full lineage searchable by compliance team

AgentCore Key Components

  • Runtime: serverless microVM, deploys any framework (Strands, LangChain, etc.) via Docker or zip upload; ECR → ECS under the hood; provides an invocation endpoint
  • Gateway: inbound auth stops here (Cognito by default); agent connects to Lambda/API Gateway/MCP tools without managing service auth directly
  • Memory — short-term: message history + state per session; configurable retention (1–365 days)
  • Memory — long-term: async extraction module pulls summaries/semantics/preferences from short-term store; agents retrieve via semantic search on demand

Multi-Agent Architecture: 8 Specialized Agents

1
Perception Agent
Parses incoming transaction; retrieves KYC data for sender + receiver from enterprise systems via Gateway. Flags entity risk status. Calculates derived features (device used, amount deviation). Not LLM-powered — it's a Python agent using tools.
2
Context Agent
Enriches with 60-day transaction history from DynamoDB via Gateway tools (Lambda functions). Computes behavioral features: rolling averages, frequency patterns, flagged transactions. Stores context in AgentCore Memory for downstream agents.
3
News Herald Agent
Fetches real-time adverse media: sanctions lists (OFAC), geopolitical news, negative press about involved parties. Uses Bedrock embeddings to write findings to Knowledge Base for memory across sessions. LLM-powered with web browser tool.
4
Reasoning Agent (core)
Combines all context (KYC + history + news) and applies LLM reasoning. Outputs: hypothesis, indicators, detailed explanation, confidence score (0–1). Uses Claude on Bedrock with guardrails to prevent hallucination. All reasoning retrieved from AgentCore Memory.
5
Risk Assessor Agent
Connects to existing classical ML risk model via API. Not LLM-powered — Python agent that calls the org's internal classification/clustering model. Adds legacy ML score as additional context for the action agent. Key: preserves existing compliance models.
6
Action Agent
Executes decisions based on risk score: score <30 → straight-through processing; 30–50 → notify + monitor; >50 → freeze + generate SAR. Tools: notification service, transaction freeze API, S3 SAR report writer.
7
Audit Agent
Collects all agent outputs + decisions + lineage. Writes to OpenSearch Serverless. Provides explainability for compliance team: "why was this flagged?" is fully traceable through each agent's reasoning.
8
Learning Agent
Writes risk assessment results back to DynamoDB as feedback loop. Patterns that were evaluated feed into the context for future similar transactions. Makes the system progressively smarter without retraining.

Risk Score → Action Tiers

  • Score < 30: Low risk → straight-through processing (automated approval)
  • Score 30–50: Medium risk → notify ops team, monitor closely
  • Score 50–80: High risk → freeze transaction, generate SAR, escalate
  • Repeated pattern flagged: mandatory SAR filing regardless of score

Layered Build Approach

  • Layer 1 (Transaction Monitoring): Perception → Context → Reasoning → Action → Audit. Core pipeline. Kinesis ingestion → DynamoDB history → Reasoning with guardrails → SAR to S3.
  • Layer 2 (Adverse Media): Add News Herald Agent. Real-time news + sanctions from Knowledge Base fed into Context Agent. AgentCore Memory prevents re-fetching same data across sessions.
  • Layer 3 (Proactive Risk): Add Risk Assessor (classical ML integration) + Learning Agent (feedback loop). Makes system reactive → proactive. DynamoDB updated with each decision.
Pattern: Build iteratively. Each layer adds intelligence without breaking the previous one. This is the correct way to introduce agents to an existing compliance stack.

Reasoning Agent Code Pattern

from strands import Agent
from strands_tools import use_aws

reasoning_agent = Agent(
    name="reasoning_agent",
    model=bedrock_model,  # Claude on Bedrock
    system_prompt="""You are a financial transaction analyst.
    Analyze the provided transaction context and produce:
    - hypothesis: what this transaction might be
    - indicators: list of risk signals
    - explanation: detailed reasoning
    - confidence: float 0-1
    - next_steps: recommended action
    Always retrieve prior context from memory before reasoning.""",
    tools=[use_aws]  # can query Bedrock KB for sanctions/memory
)

# AgentCore Memory context passed via message history
result = reasoning_agent(
    f"Analyze transaction: {transaction_context}",
    session_context=session_memory.get_context()
)

AgentCore Gateway Pattern

# Lambda function as a tool via Gateway
@agent.tool_plain
def get_historical_transactions(
    account_id: str,
    days_back: int = 60
) -> dict:
    """Get transaction history for an account from the last N days via DynamoDB."""
    import httpx
    # Gateway handles auth; agent just calls the tool URL
    response = httpx.get(
        f"{GATEWAY_URL}/tools/get-transactions",
        params={"account_id": account_id, "days": days_back},
        headers={"Authorization": cognito_token}
    )
    return response.json()
  • Gateway handles inbound Cognito auth — the agent just calls the tool endpoint
  • Adding a new data source = new Lambda function + new tool registration. No agent code changes.

Key Takeaways for Portfolio Projects

  • The agent architecture is fully modular — you can implement just the Transaction Monitoring layer (Agents 1, 2, 4, 6, 7) as a standalone portfolio project
  • Use AgentCore Memory to maintain context across a session — don't pass huge payloads between agents
  • Mix LLM-powered and Python-powered agents: not every agent needs an LLM. Rule engines and ML models can be agents too.
  • AgentCore observability gives you tool call tracing, timing, and lineage for free — no extra setup needed
  • Different models for different tasks: Claude for reasoning, Nova for classification/summarization
