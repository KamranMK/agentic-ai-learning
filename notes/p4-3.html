Level 1: Basic Agent with Shell Tool

from strands import Agent
from strands_tools import shell

agent = Agent(
    system_prompt="You are a helpful coding assistant.",
    tools=[shell]  # access to the environment it's running in
)

agent("What Python packages are installed?")
  • Shell tool = the agent can run any bash command in its environment
  • Foundation for self-extension: if it can write files, it can write tool definitions

Level 2: Self-Extending Agent (Hot Tool Reloading)

from strands import Agent
from strands_tools import shell

agent = Agent(
    system_prompt="""Create tools for yourself and start using them.
    Store tools in the ./tools/ directory using the @tool decorator pattern.
    Tools will be automatically loaded from that directory.""",
    tools=[shell],
    load_tools_from_directory="./tools",  # watches directory for new .py files
)

agent("Create useful tools and demonstrate them.")
  • Agent uses the shell to write Python files to ./tools/
  • Strands hot-reloads the new tool files without restarting
  • Agent immediately starts using its newly created tools in the same run
  • In the demo: agent creates calculator.py, text_analyzer.py, unit_converter.py, etc., then uses them
Why this matters: Agents can expand their own capabilities in response to what a task requires — no human needs to write new tool code.

Level 3: Self-Modifying System Prompt

import os
from strands import Agent
from strands_tools import shell, environment_tool

# System prompt lives in an env var (or DynamoDB, S3 — any persistent store)
def get_agent():
    system_prompt = os.environ.get("AGENT_SYSTEM_PROMPT", "You are a helpful assistant.")
    agent = Agent(
        tools=[shell, environment_tool],  # environment_tool can read/write env vars
    )
    agent.system_prompt = system_prompt  # re-set every invocation
    return agent

while True:
    user_input = input("You: ")
    agent = get_agent()  # rebuilds with latest system prompt
    agent.messages = []  # clear conversation history
    agent(f"Update your system prompt if needed, then respond: {user_input}")

Three required steps for a self-modifying system prompt:

1 Dynamic construction: system prompt has a dynamic portion loaded from external storage (env var, DynamoDB, S3)
2 Self-modification: agent is given a tool to update that storage (environment_tool writes to env; swap for DynamoDB write in production)
3 Persistence: changes survive across invocations; sub-agents spawned later also read the updated prompt
Sub-agent gotcha: If you create sub-agents programmatically, they must also load from the same persistent store to see the updated system prompt. This is why persistence matters.

The Core 3-Step Loop


  ┌─────────────────────────────────────────────────────┐
  │   1. RETRIEVE   →   2. MODIFY   →   3. PERSIST     │
  │                                                     │
  │  Load latest       Agent adapts    Save changes     │
  │  system prompt,    behavior        to external      │
  │  memories,         (new tools,     storage (S3,     │
  │  knowledge         new prompt,     DynamoDB,        │
  │                    new agents)     file system)     │
  └─────────────────────────────────────────────────────┘
       ↑                                     │
       └─────────────── on next invocation ──┘
  • This loop is the foundation of all self-evolving agent patterns
  • Can be triggered by: human messages, scheduled events, other agents, even the agent itself

Level 4: Memory via Vector Store

from strands import Agent
from strands_tools import shell, retrieve_from_kb, store_in_kb

agent = Agent(
    system_prompt="You are a personal assistant. Remember everything about the user.",
    tools=[shell, retrieve_from_kb, store_in_kb]
)

# Workflow per interaction:
# 1. Pre-fill context: directly call retrieve to add memories to agent.messages
agent.tool.retrieve_from_kb(query=user_input)  # fills agent's context

# 2. Run agent (now has memory context in messages)
result = agent(user_input)

# 3. Store interaction for future sessions
agent.tool.store_in_kb(
    title="conversation_history",
    content=f"User: {user_input}\nAssistant: {result}"
)
  • retrieve_from_kb tool called via agent.tool.X() = direct programmatic call, bypasses LLM decision
  • This pre-fills agent.messages (the context window) with relevant memories before the agent runs
  • Agent can also call retrieve_from_kb dynamically during a run if it decides it needs more memory
Memory options: Bedrock Knowledge Base (semantic, distributed) · S3 Vectors · Mem0 · AgentCore Memory (managed, recommended for prod) · ChromaDB (local, fast, not distributed) · Markdown journal files (no dependency, no semantic search)

Level 5: Meta Agents — Dynamic Sub-Agent Creation

  • The orchestrator agent has a tool called use_agent that creates new sub-agents at runtime
  • The LLM decides when to spin up a sub-agent and what system prompt / tools / model to give it
  • No engineer pre-defines the sub-agent — the orchestrator defines it based on what the task needs
from strands import Agent, tool
from strands_tools import shell, think, swarm, graph, journal

@tool
def use_agent(
    system_prompt: str,
    tools: list[str],
    model_provider: str,
    task: str
) -> str:
    """Create and invoke a new specialized sub-agent for a specific task."""
    sub_agent = Agent(
        system_prompt=system_prompt,
        model=resolve_model(model_provider),
        tools=resolve_tools(tools)
    )
    return str(sub_agent(task))

# Meta-agent orchestrator
meta_agent = Agent(
    system_prompt="You are an orchestrator. Use use_agent to create specialized helpers.",
    tools=[shell, use_agent, think, swarm, graph, journal]
)

meta_agent("Research and write a comprehensive report on quantum computing trends.")
  • Sub-agents each get their own fresh 1M-token context window — expands effective context dramatically
  • Fire-and-forget: orchestrator can spin up sub-agents in parallel without waiting for results
  • Results propagated back via journal (file), shared memory, or return value

Meta-Agent Tool Primitives

use_agent
Create a new sub-agent with custom system prompt, tools, and model. Can run sync (wait for result) or async (fire and forget). Each sub-agent gets a fresh 1M-token context.
think
Chain-of-thought tool: invokes the agent N times, each building on the previous response. Like "think deeper" — useful for research and code review. Each invocation refines understanding.
swarm
Create a peer group of agents with a shared context object. Agents coordinate without a central boss. Each agent can update the shared context that others can read. Useful for multi-perspective analysis.
graph
Create a directed graph of agents — define which agents communicate with which. Provides predictable, controlled information flow. Can include loops (agent can re-process until condition met).
journal
File-system-based shared scratchpad. All sub-agents read/write to the same markdown files. No embedding/retrieval — just direct file I/O. Works offline, no vector store dependency. Low latency for coordination.
Composability: All primitives can be mixed. A graph node can use use_agent. A swarm member can use think. This is the key architectural freedom of meta-agents.

Context Expansion Pattern


Orchestrator (1M token context)
├── use_agent → Sub-agent A (1M fresh context)
│                └── use_agent → Sub-agent A1 (1M context)  ← fire & forget
│                └── use_agent → Sub-agent A2 (1M context)  ← fire & forget
├── use_agent → think(10x) → Sub-agent B (1M × 10 iterations of refinement)
└── swarm([C, D, E]) → shared context between 3 peer agents
  • Worst case token budget: 4M tokens for the diagram above (4 independent sub-agents)
  • Sub-agent results returned to orchestrator via journal or direct return
  • Parallel execution: orchestrator spins up sub-agents in parallel, collects results asynchronously

Level 6: Deploy to AgentCore Runtime

from bedrock_agentcore import BedrockAgentCoreApp
from bedrock_agentcore.memory import MemoryClient
from strands import Agent

app = BedrockAgentCoreApp()
memory = MemoryClient()  # AgentCore managed memory (provisions automatically)

@app.entrypoint
def handle(payload):
    # Retrieve relevant memories
    context = memory.retrieve(query=payload["message"])

    agent = Agent(
        system_prompt="You are an autonomous assistant.",
        tools=[shell, use_agent, think, swarm, graph, journal]
    )
    agent.messages = context  # pre-fill with memories

    result = agent(payload["message"])

    # Store interaction
    memory.store(content=f"User: {payload['message']}\nAgent: {result}")
    return {"response": str(result)}

if __name__ == "__main__":
    app.run(port=8080)

# Deploy (3 commands):
# agentcore config     → generates Dockerfile
# agentcore launch     → builds container, pushes to ECR, deploys to Runtime
# agentcore invoke --payload '{"message": "What did I work on last week?"}'
  • agentcore dev: run locally in a container that matches the production interface — like SAM CLI for agents
  • Supports CDK and Terraform for CI/CD pipeline integration
  • AgentCore provisions the Bedrock Knowledge Base for memory automatically on agentcore launch
  • Tools written to ./tools/ are not persistent across container restarts — persist to S3 and re-download on startup

Controlling Self-Evolving Agents (AgentCore Policies)

  • Self-evolving agents need bounds — "evolve, but stay within these constraints"
  • AgentCore Policies: write constraints in natural language → auto-generates a formal policy (like IAM)
  • Built on CEDAR (open-source policy language from AWS) — formally provable constraints
  • Policies are outside the agent container — even if the agent modifies its own code, policies can't be overridden
Key principle: Policies define the sandbox. The agent can evolve freely within it, but can't break out of it. This is how you ship self-evolving agents responsibly to production.

Key Takeaways

  • Self-evolving agents follow a 3-step loop: Retrieve → Modify → Persist. Get this loop right and everything else follows.
  • load_tools_from_directory + shell tool = agents that build their own capabilities with zero restarts
  • Store system prompts in external storage (DynamoDB, S3, env vars) — never hardcode them in agents that need to evolve
  • The use_agent tool is the key meta-primitive: orchestrators don't need pre-defined sub-agents — they create the right one for the task
  • Context expansion: each sub-agent gets a fresh 1M-token window — unlimited effective context for complex tasks
  • Use AgentCore Policies to define boundaries before shipping self-evolving agents to production
  • Human-in-the-loop is still valuable — review what the agent builds (especially new tools) before giving it broader access
