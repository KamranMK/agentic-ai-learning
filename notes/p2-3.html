Why Observability Matters for Agents

  • Agents are compound systems: vector stores + models + evals + orchestration — many failure points
  • Agents are non-deterministic: same input ≠ same output; you can't predict failures in advance
  • Multiple accountability layers: model? framework? orchestration? tool call? Hard to debug without traces
  • Cost can spiral: every bad tool call loop consumes tokens; you need visibility to catch this
  • Rule: Start with observability from the first prototype, not as an afterthought

Four Production Challenges

  • Reliability: hallucinations, wrong tool calls, goal drift — need evals to catch these
  • Complexity: when something goes wrong in a multi-agent system, where exactly did it fail?
  • Cost: token consumption per model call; bad loops can cost 10–100× what they should
  • Security: prompt injection, PII exposure, jailbreak attempts — need input/output monitoring

Newsletter Agent Architecture (Demo)

  • Orchestrator agent + 1 specialist "deep analysis" sub-agent (agent-as-a-tool pattern)
  • Tools: fetch current date, fetch latest news, extract article links, publish newsletter (email)
  • Two operation modes: browse mode (Q&A with agent) and publish mode (triggers full newsletter workflow)
  • Agent manages its own EventBridge schedule — can schedule itself to run daily
  • Custom memory strategy: tracks URLs of previously sent articles to prevent duplicates across sessions
Pattern: Give agents only the tools they need — not everything. It's cheaper, faster to reason, and easier to control. Agent specialization > agent omniscience.

Strands + AgentCore Integration (2 lines to connect)

import os
from strands import Agent
from strands.models import BedrockModel

# Point Strands telemetry at Datadog's OTel ingestion endpoint
os.environ["OTEL_EXPORTER_OTLP_ENDPOINT"] = "https://ingest.datadoghq.com/v1/traces"
os.environ["OTEL_EXPORTER_OTLP_HEADERS"] = f"DD-API-KEY={os.environ['DD_API_KEY']}"

agent = Agent(
    model=BedrockModel(model_id="us.anthropic.claude-sonnet-4-5-20251001-v1:0"),
    system_prompt="You are a news curator assistant."
)

# All traces (model calls, tool calls, latency, tokens) now flow to Datadog automatically
  • Strands emits OTel traces by default — no SDK changes needed
  • Two environment variables is all it takes to connect to Datadog
  • Same pattern works for any OTel-compatible backend: Langfuse, Grafana, Honeycomb, Jaeger

AgentCore Runtime Deployment

from bedrock_agentcore import BedrockAgentCoreApp

app = BedrockAgentCoreApp()

@app.entrypoint  # one decorator = deployable to AgentCore
def run_agent(payload):
    result = agent(payload["query"])
    return {"response": str(result)}

if __name__ == "__main__":
    app.run(port=8080)

# Deploy:
# agentcore config   (generates docker config)
# agentcore launch   (builds container, pushes to ECR, deploys)
# agentcore invoke --payload '{"query": "What is the AWS news today?"}'
  • AgentCore Runtime = serverless microVM; sessions up to 8 hours; scales automatically
  • Also supports CDK/Terraform for CI/CD pipeline integration
  • AgentCore Memory: short-term (per-session) + long-term (semantic extraction) available with a few extra lines

What Datadog Shows (LLM Observability Dashboard)

  • Application overview: error rate, duration p50/p95, estimated token cost, LLM calls per hour
  • Trace explorer: every agent invocation with full span tree — see exactly which tool was called and when
  • Cost breakdown: total cost + per-model breakdown; spot which LLM calls are expensive
  • Cluster view: topic clustering — what is your agent actually being asked? Spot unexpected topics
  • Tool call view: how often each tool is called; spot over-indexing on specific tools
  • Security panel: prompt injection attempts, PII exposure (auto-stripped), output toxicity

Evaluations in Datadog

  • Out-of-box evals (LLM-as-judge): failure-to-answer, goal completeness, hallucination, input toxicity, output toxicity, prompt injection detection, tool argument correctness
  • Custom evals: define your own criteria (e.g., "agent must respond in brand voice") — LLM judge scores each trace
  • Choose your judge model; it classifies all incoming traces automatically in the background
# Create an experiment to compare two approaches
# 1. Add traces to a dataset (from real traffic)
# 2. Define inputs: model variants, prompt variants
# 3. Measure outputs: information accuracy, user satisfaction, duration, error rate, token cost
# 4. Datadog shows side-by-side comparison — pick the best performing configuration
Recommendation: Choose the most capable model first, establish a performance baseline, then experiment with cheaper models to find where you can save cost without hurting quality.

Well-Architected Framework: GenAI Lens (Key Points)

  • Operational excellence: collect metrics/user feedback/performance data; implement guardrails to detect policy violations; monitor tool call success + latency; track cost per workflow/user/model
  • Security: detect prompt injection, jailbreak attempts, PII exposure; Datadog strips PII before ingesting all traces
  • Reliability: monitor answer quality (hallucination rate, goal completeness); alert on threshold breaches
  • Cost optimization: measure inference efficiency; identify which model can be downsized for which task

Trace View: Debugging a Bad Agent Run

  • Each agent run = a root span with child spans for every LLM call and tool call
  • Click any span to see: input, output, token count, cost, duration, errors
  • Playground view: test the sub-agent's prompt in isolation to iterate without running the full pipeline
  • Example caught from demo: user asked "drop these tables" → Datadog flagged prompt injection in the trace

Key Takeaways

  • Start with observability on day one. Retrofitting it is painful and you'll miss the debugging data from your earliest experiments.
  • Strands emits OTel automatically — connect to Datadog/Langfuse/Grafana with 2 env vars
  • Use LLM-as-judge evals for quality monitoring — especially for multi-agent pipelines where output quality is hard to measure manually
  • AgentCore Runtime + Memory makes going from prototype to production a 3-command process
  • Agents should be purpose-built with the minimum tool set for their task — reduces cost, improves reasoning quality, easier to debug
  • Agent scheduling: agents can manage their own EventBridge schedules — enables fully autonomous workflows
